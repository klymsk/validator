## 1. Діаграма компонентів (Component Diagram)

Система Nested Schema Validator розділена на незалежні модулі відповідно до принципу Single Responsibility. Кожен модуль виконує одну логічну задачу, що спрощує підтримку та тестування системи.

### Високорівневий поділ на модулі:
1. **Input Parser**

Точка входу для обробки файлів.

Відповідає за:

- зчитування JSON-файлів
- перевірку коректності формату
- перетворення тексту у внутрішні структури даних

Input Parser отримує:

- файл схеми
- файл даних

Результат:

- структуровані об'єкти Schema та Data


---

2. **Data Model**

Модуль представлення даних у пам'яті.

Відповідає за:

- зберігання схеми
- зберігання даних
- представлення вкладених структур

Data Model містить:

- структуру Schema
- структуру Data
- ValidationResult


---

3. **Validation Engine**

Основний модуль системи.

Відповідає за:

- перевірку типів даних
- перевірку обов'язкових полів
- перевірку вкладених об'єктів
- перевірку масивів

Validation Engine отримує:

- Schema
- Data

Результат:

- ValidationResult


---

4. **Report Generator**

Модуль формування результатів.

Відповідає за:

- створення звітів
- форматування повідомлень про помилки
- вивід результатів

Report Generator отримує:

- ValidationResult

Результат:

- текстовий звіт


---

### Напрямок потоку даних

Потік даних у системі є послідовним:

Користувач  
→ Input Parser  
→ Data Model  
→ Validation Engine  
→ Report Generator  
→ Користувач

## 2. Опис інтерфейсів та методів (Class Diagram)

Архітектура системи Nested Schema Validator побудована за принципом модульності та розділення відповідальностей. Кожен компонент системи виконує окрему функцію та взаємодіє з іншими компонентами через визначені інтерфейси.
Основна взаємодія між компонентами реалізована через передачу структур даних між модулями.

### Логіка взаємодії компонентів
## Основний процес роботи системи складається з декількох етапів.

1. **Зчитування та підготовка даних**

Після запуску програми користувач передає:

- файл схеми
- файл даних

Модуль InputParser виконує::

- зчитування файлів
- перевірку коректності JSON
- перетворення даних у внутрішні структури

Результат:

- структуровані об'єкти Schema та Data

Ці об'єкти передаються до ValidationEngine.

---

2. **Валідація даних**
Модуль ValidationEngine є центральним елементом системи.
Він отримує:

- Schema
- Data

ValidationEngine виконує:
- перевірку відповідності типів
- перевірку required полів
- перевірку вкладених об'єктів
- перевірку масивів

ValidationEngine використовує рекурсивний алгоритм перевірки вкладених структур.
Результатом є об'єкт ValidationResult.

---

3. **Формування результату**

Модуль ReportGenerator отримує ValidationResult.
ReportGenerator виконує:

- формування текстового звіту
- форматування повідомлень
- вивід результату користувачу

---

## Основні абстракції системи
### Архітектура системи базується на декількох ключових абстракціях.

--- 

### Schema
Schema представляє структуру даних, яка використовується для перевірки.
Schema містить:
- список полів
- типи полів
- вкладені структури
- required поля
Schema використовується ValidationEngine для перевірки даних.
### Data
Data представляє вхідні дані користувача.
Data містить:
- значення полів
- вкладені об'єкти
- списки значень
Data передається у ValidationEngine.
### ValidationResult
ValidationResult представляє результат перевірки.
ValidationResult містить:
- статус перевірки
- список помилок
ValidationResult передається у ReportGenerator.

## 3. Опис структур даних та обґрунтування

Для забезпечення ефективної роботи системи перевірки даних було обрано структури даних, які забезпечують швидкий доступ до інформації, простоту реалізації та зрозумілу логіку обробки.

---

### Хеш-таблиця (Dictionary / Map)

**Місце використання:**  
Використовується для зберігання схеми даних (Schema) та записів даних (DataRecord).  
Ключем є назва поля, значенням — тип даних або конкретне значення.

**Обґрунтування:**  
Хеш-таблиця забезпечує асимптотичну складність доступу **O(1)** для операцій пошуку поля за назвою.  
Оскільки під час перевірки даних система багаторазово звертається до полів за їх іменами, використання словника значно прискорює процес валідації.

Ця структура також дозволяє:

- легко перевіряти наявність поля
- швидко отримувати тип поля
- зручно працювати з JSON-даними

Приклад використання:

Schema:

```json
{
"name": "→ string"
"age": "→ int"
"email": "→ string"
}
```

DataRecord:
```json
{
"name": "→ Alex"
"age": "→ 20"
"email": "→ example@gmail.com"
}
```

---

### Список (List)

**Місце використання:**  
Використовується для зберігання:

- списку помилок перевірки
- списку результатів перевірки
- набору записів даних

**Обґрунтування:**  
Список дозволяє ефективно зберігати послідовність елементів у порядку їх додавання.

Операція додавання нового елемента має складність **O(1)**, що важливо при накопиченні помилок під час перевірки.

Список також дозволяє:

- легко перебирати результати перевірки
- формувати звіти
- зберігати історію перевірок

Приклад:

```json
{
"missing field": "→ email"
"Invalid type": "→ age"
}
```


---

### Рядки (String)

**Місце використання:**  

Використовуються для представлення:

- назв полів
- типів даних
- повідомлень про помилки
- шляхів до файлів

**Обґрунтування:**  

Рядки є природним способом представлення текстових даних.

Використання рядків дозволяє:

- легко працювати з JSON-файлами
- формувати зрозумілі повідомлення про помилки
- обробляти вхідні параметри програми


---

### Об'єкти (Classes)

**Місце використання:**  

Використовуються для представлення основних компонентів системи:

- Schema
- DataRecord
- ValidationResult
- ValidationEngine

**Обґрунтування:**  

Використання класів дозволяє логічно розділити функціональність програми.

Кожен клас відповідає за власну частину роботи:

- Schema — зберігає структуру даних
- DataRecord — зберігає дані
- ValidationResult — зберігає результат перевірки
- ValidationEngine — виконує перевірку

Це забезпечує:

- зрозумілу архітектуру
- простоту підтримки
- можливість розширення

---

### JSON-структури

**Місце використання:**  

Використовуються для зберігання:

- схеми даних
- даних для перевірки

**Обґрунтування:**  

JSON є стандартним форматом обміну даними.

Використання JSON дозволяє:

- легко створювати тестові дані
- зручно редагувати файли
- забезпечити сумісність з іншими системами

---

### Взаємодія структур даних

Структури даних взаємодіють наступним чином:

```bash
JSON файли
    |
    |
Dictionary (Schema)
Dictionary (DataRecord)
    |
    |
ValidationEngine
    |
    |
List (Errors)
    |
    |
Report
```


Schema визначає структуру даних.

DataRecord містить дані.

ValidationEngine перевіряє дані.

Errors містить список помилок.
